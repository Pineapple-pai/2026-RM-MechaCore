# 舵轮仓库说明

本仓库是舵轮底盘相关代码的整理版，重点保留了底盘控制、云台到底盘通信、功率管理、运动学解算以及部分调试与 UI 逻辑。

这份代码更适合作为阅读、复用和二次整理的代码仓库，而不是一个开箱即用的完整嵌入式工程。当前目录主要包含 `Chassis/User` 和 `Gimbal/User` 两部分用户层代码，用来展示舵轮底盘与云台侧之间的协作逻辑。

如果你需要完整工程项目，请查看原始仓库分支：

- 底盘/云台整工程之一：<https://github.com/Pineapple-pai/H_SG_Gimbal/tree/H_Gimbal>
- 底盘整工程之一：<https://github.com/Pineapple-pai/H_SG/tree/SG_New>

## 这份仓库里有什么

### 1. `Chassis/User`

这一部分是舵轮底盘主控制代码，核心职责是：

- 接收云台板发来的控制指令
- 根据模式生成底盘速度目标
- 做舵轮逆运动学解算
- 更新转向电机和驱动电机控制量
- 进行功率限制、超电协同和状态上报

重点目录和文件如下：

- `Task/ChassisTask.cpp`
  - 底盘主任务。
  - 包含多个模式处理器，如通用模式、跟随模式、自旋模式、键鼠模式、停止模式。
  - 主链路基本是：目标生成 -> 运动学解算 -> 反馈滤波 -> PID 更新 -> CAN 下发。

- `Algorithm/ChassisCalculation/StringWheel.hpp`
  - 四舵轮底盘的运动学相关实现。
  - 包含正向/逆向运动学计算，是四模块舵轮控制的核心基础。

- `Algorithm/ChassisCalculation/DualSteerIK.hpp`
  - 两主动舵轮构型的逆运动学解算。
  - 适合两舵两全之类的底盘布置。
  - 做了“就近转位”处理：若目标舵角跨越过大，会翻转轮速方向，缩短转向路径。

- `Task/PowerTask.cpp`
  - 底盘功率控制任务。
  - 负责结合裁判系统功率限制、超电状态、能量缓冲区状态，动态调整底盘可用功率。
  - 同时包含调试输出和功率状态快照上报逻辑。

- `Task/CommunicationTask.*`
  - 负责底盘与云台之间的数据通信。
  - 底盘侧主要读取云台发来的模式、摇杆量、旋转速度、UI 状态等数据。

- `BSP/Motor/`
  - 电机驱动抽象，包含 DJI、LK、DM 等不同电机接口适配。

- `HAL/CAN/`
  - CAN 抽象层及实现。
  - 通信任务和电机控制最终都通过这里向下接硬件总线。

- `APP/UI/`
  - 与裁判系统/UI 刷新相关的逻辑。
  - 用于显示底盘状态、功率状态、摩擦轮状态等信息。

### 2. `Gimbal/User`

这一部分不是舵轮底盘本体，但它是舵轮系统的重要上游输入源，负责把操作意图和部分状态同步给底盘。

重点职责：

- 读取遥控器/键鼠输入
- 管理云台与发射机构模式
- 通过 CAN 把底盘控制量发给 `Chassis`
- 维护视觉、热量和部分交互逻辑

重点目录和文件如下：

- `Task/CommunicationTask.cpp`
  - 这是云台发往底盘的关键入口之一。
  - 负责打包底盘方向量、模式位、UI 状态，并通过 CAN 分帧发送到底盘。
  - 键鼠模式下也会在这里生成 `WASD` 对应的平移目标和旋转开关逻辑。

- `Task/GimbalTask.*`
  - 云台控制主任务。

- `Task/ShootTask.*`
  - 发射机构控制逻辑。
  - 包含摩擦轮、拨盘、热量和状态机相关实现。

- `APP/Heat_Detector/`
  - 发射热量控制和发射事件检测逻辑。

- `APP/Mod/`
  - 遥控器模式、控制器抽象等模块。
  - 底盘模式最终很多就是从这里派生并同步到 `Chassis` 侧。

## 代码主流程

从系统协同角度看，这份仓库的主要执行链路可以概括为：

1. 云台侧读取遥控器、键鼠或视觉输入。
2. `Gimbal/User/Task/CommunicationTask.cpp` 将底盘需要的数据打包，通过 CAN 发给底盘。
3. 底盘侧通信模块接收这些模式位和目标量。
4. `Chassis/User/Task/ChassisTask.cpp` 按当前模式生成 `vx / vy / vw` 目标。
5. 舵轮运动学模块计算每个舵向电机和驱动电机的目标值。
6. PID、滤波和功率控制模块共同约束输出。
7. 最终通过 CAN 向各个电机下发控制指令。

## 适合从哪里开始读

如果你第一次看这份仓库，建议按下面顺序：

1. 先看 `Chassis/User/Task/ChassisTask.cpp`
2. 再看 `Chassis/User/Algorithm/ChassisCalculation/StringWheel.hpp`
3. 如果你的底盘是两主动舵轮，再看 `Chassis/User/Algorithm/ChassisCalculation/DualSteerIK.hpp`
4. 然后看 `Chassis/User/Task/PowerTask.cpp`
5. 最后看 `Gimbal/User/Task/CommunicationTask.cpp`

这样可以先建立“控制链路”，再看“算法细节”和“协同逻辑”。

## 这份仓库没有完整包含的内容

这份仓库当前更偏向用户层代码整理，通常不应默认理解为完整可编译工程。和完整项目相比，可能缺少或不完整的内容包括：

- IDE 工程文件
- 芯片启动文件和完整链接脚本
- CubeMX 生成内容或完整 BSP 初始化工程
- 某些板级配置、第三方库或中间件集成
- 与原始仓库分支完全同步的工程目录结构

因此，如果你的目标是：

- 直接编译下载到板子
- 对照完整硬件资源配置
- 查完整工程依赖关系

请优先回到上面的两个原始 GitHub 分支查看。

这份仓库适合用于：

- 阅读舵轮底盘控制架构
- 参考底盘与云台之间的 CAN 协议组织方式
- 复用舵轮逆运动学与功率控制思路
- 抽取某些模块到新工程中做二次开发

如果只是想快速找到“舵轮控制核心”，优先看：

- `Chassis/User/Task/ChassisTask.cpp`
- `Chassis/User/Algorithm/ChassisCalculation/StringWheel.hpp`
- `Chassis/User/Algorithm/ChassisCalculation/DualSteerIK.hpp`
- `Chassis/User/Task/PowerTask.cpp`
